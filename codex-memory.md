# Codex 的记忆：给 Agent 小白的一页纸

> 你懂 Agent 是什么，但不一定看过 Codex 源码——这篇只讲**产品行为**和**设计逻辑**。

---

## Agent 为什么需要「记忆」

每次新开一个 Agent 会话，模型默认**不记得**你上周说过什么、项目里踩过什么坑。

Codex 的记忆系统要做的事很简单：

1. **后台**把历史会话里值得保留的东西抽出来、整理好  
2. **前台**让 Agent 在需要时能找到这些东西，少重复犯错、少重复解释

它不是聊天记录全文检索，也不是给模型装一个「无限上下文」。它是一套**读写分开**的流水线：**写得慢、读得轻**。

---

## 30 秒总览

```
你用完 Codex 开会话
        ↓
【写】后台 Phase 1：每个会话抽一份笔记 → 先进内部仓库
        ↓
【写】后台 Phase 2：挑最重要的 N 份，合并成几本「手册」→ 落到文件夹
        ↓
【读】下次开会话：薄索引自动塞进 Agent；厚的自己搜
```

记忆开关可以拆开：**可以只读不写，也可以读写都开**（具体看配置）。

---

## 写记忆：两阶段流水线

### Phase 1 —— 会话笔记（逐条处理）

会话结束后（或空闲一段时间后），后台会对**符合条件的会话**各跑一次小模型，产出两样东西：

- **详细笔记**：这个会话里用户偏好、踩坑、可复用做法……  
- **短摘要**：一两段话概括这场会话干了啥

此时笔记还在**内部数据库**里，Agent **还看不到**。  
同一场会话如果后来又更新了，会**覆盖**旧笔记，不会无限堆版本。

### Phase 2 —— 合并成手册（全局整理）

Phase 1 攒了一批笔记后，后台再做一次「编辑部」工作：

1. **挑选**：从所有笔记里选出最多 **256 条**、且在 **30 天**新鲜度窗口内的（默认参数）  
2. **同步文件**：把选中的写成 `rollout_summaries/`（每场会话一篇短文）和 `raw_memories.md`（原始笔记合集）  
3. **合并 Agent**：读「上次手册 vs 这次新增了啥」的差异，更新下面要说的**三本成品**

合并 Agent 在**隔离环境**里跑：不能上网、不能再开子 Agent、也不能递归触发记忆写入——避免自己改自己、越改越乱。

**掉出 Top-256 的笔记**会从文件夹里**删掉**（但可能还在数据库里待淘汰）。所以磁盘上的记忆是**精选集**，不是全量归档。

## 读记忆：Agent 实际看到什么

记忆住在 `~/.codex/memories/`。**运行时 Agent 只读这个文件夹**，不直接查内部数据库。

### 文件夹全景

```
memories/
├── memory_summary.md          # 薄索引 → 每次自动注入 Agent
├── MEMORY.md                  # 厚手册 → 主查询入口（自己搜）
├── raw_memories.md            # Phase 2 输入：Top-N 原始笔记合集（给合并 Agent）
├── rollout_summaries/         # 每场会话一张短卡片（recap + 元数据）
├── skills/                    # 可复用流程包（SKILL.md + 可选脚本）
├── extensions/                # 外来信号入口
│   └── ad_hoc/notes/          # 用户/Agent 提交的「小纸条」，等下轮合并
├── phase2_workspace_diff.md   # 临时：本轮变更 diff，合并完即删
└── .git/                      # 内部 baseline，用户一般不用管
```

**谁读、谁写：**

| 路径 | 读者 | 写者 |
|------|------|------|
| `memory_summary.md` | 运行时 Agent（注入） | 合并 Agent |
| `MEMORY.md` | 运行时 Agent（搜索） | 合并 Agent |
| `raw_memories.md` | 合并 Agent | **代码**机械同步 |
| `rollout_summaries/` | Agent 按需 / 合并 Agent | **代码**机械同步 |
| `skills/` | Agent 按需 | 合并 Agent |
| `extensions/ad_hoc/notes/` | 合并 Agent | **运行时 Agent**（仅用户明确要求时） |
| `phase2_workspace_diff.md` | 合并 Agent | **代码**生成，成功后删除 |

`raw_memories.md` 和 `rollout_summaries/` 是 **Phase 2 的「编辑部原料」**，不是给 Agent 每次从头翻的全库；Agent 正常路径仍是 **summary → MEMORY.md → 按需开摘要**。

### 三本成品，分工不同

| 文件 | 像什么 | Agent 怎么用 |
|------|--------|--------------|
| **memory_summary.md** | 封面 + 目录 | **每次会话自动注入**，不用自己打开；截断后大约一两千字量级 |
| **MEMORY.md** | 厚手册 | **不自动注入**；需要时按关键词搜索（`rg` / `grep` 或专用搜索工具） |
| **skills/** | 可复用的「操作流程包」 | 只有手册指向时才打开；每个 skill 是一个文件夹 + 入口说明 |

另外还有中间文件 `rollout_summaries/`（每场会话摘要）——一般等 `MEMORY.md` 指路再去读。

**为什么分薄厚？** 上下文窗口有限。每次先把「你是谁、常做啥、去哪查」塞给 Agent；细节等它判断**真的需要**再翻。

### 检索机制：渐进式 + 文本搜索，没有向量

Codex 记忆**没有** embedding、**没有**向量库、**没有**语义相似度排序。

标准路径是 Agent 自己执行的**渐进式发现**：

1. 看已注入的 **summary**，判断这事要不要查记忆  
2. 提几个关键词，**搜 MEMORY.md**  
3. 手册里若指向某场会话摘要或某个 skill，再打开 **1～2 个**相关文件  
4. 若要精确命令、报错原文，再去原始会话记录里搜  

建议整个 memory pass 控制在 **几步之内**，避免把记忆目录翻个底朝天。

可选地，可以开启专用工具 `memories/search`、`read`、`list`——本质仍是**子串匹配**，不是 AI 语义搜索。

## 内部数据库：`memories_1.sqlite`

Agent 碰不到，但理解流水线需要知道它干什么。

位置：`~/.codex/` 下（与 `state_5.sqlite` 并列；后者管**会话列表/元数据**，前者管**记忆流水线**）。

| 表 | 存什么 |
|----|--------|
| **`stage1_outputs`** | 每场会话一行：详细笔记、短摘要、生成时间、`usage_count` / `last_usage`（引用反馈） |
| **`jobs`** | Phase 1 / Phase 2 后台任务状态（认领、租约、成功/失败） |

**和文件夹的关系：**

```
Phase 1 写完 → 只在 SQLite
Phase 2 选中 Top-N → 代码把对应行同步到 rollout_summaries/ + raw_memories.md
                  → 合并 Agent 再写 summary / MEMORY / skills
掉出 Top-N     → 文件夹里删掉，行可能还在直到 prune
```

所以：**文件夹 = 当前精选集的「可读面」；SQLite = 全量笔记 + 任务调度 + 用量统计的「写面」**。

## 引用闭环（Citation）

Agent 用了记忆后，规范要求它在**回复最末尾**附一个 `<oai-mem-citation>` 块。这是记忆系统唯一的**自动反馈通道**。

```
Agent 读 memory_summary / MEMORY.md / rollout_summaries / skills
        ↓
正常作答，末尾附引用块
        ↓
系统解析 rollout_ids（会话 UUID）
        ↓
SQLite：对应 stage1_outputs.usage_count += 1，更新 last_usage
        ↓
下次 Phase 2 选 Top-N 时，这条笔记排序更靠前
```

块里两段，用途不同：

| 字段 | 给谁用 | 内容 |
|------|--------|------|
| **`citation_entries`** | 展示 / 审计 | 用了哪个文件、哪几行、一句话说明怎么用上的 |
| **`rollout_ids`** | **反馈排序** | 哪些历史会话被认为有用（UUID，一行一个） |

**注意：**

- 只读 `MEMORY.md` 打**遥测**，**不写** `usage_count`；要有 `rollout_ids` 才进闭环。  
- `rollout_ids` 可从 `MEMORY.md` 或 `rollout_summaries/` 的 `thread_id` 拿到，不必先打开原始 jsonl。  
- 引用次数**帮热门笔记留座**，**不是**进文件夹的门票（见下节）。

## 几个最容易误解的点

### 1. 「引用计数」不是入场券

Agent 回复末尾可以带一个记忆引用块，里面列出用了哪些文件、哪些会话 ID。系统据此给对应笔记 **+1 使用次数**。

但这**不是**「被引用过才能进手册」：

- **进文件夹**靠 Phase 2 的 Top-256 选择  
- **使用次数**只影响**下次谁更容易留在 256 席里**

新笔记一开始都是 0 次引用，**完全可以进盘**——只要池子没挤满。  
真正麻烦的是：若 30 天内已有 **≥256 条**被引用过的笔记占满席位，新的 0 引用笔记**暂时挤不进去**，形成「要先被用才能稳住在盘里」的间接压力——不是死锁，旧笔记老化后会腾位。

### 2. 运行时 Agent 不能改手册

普通 Agent **不能**直接编辑 `MEMORY.md` 或 `memory_summary.md`。  
用户明确要求「记住这个」时，只能往 `extensions/ad_hoc/notes/` 丢一张小纸条，等下一轮合并吸收。

### 3. `memories/skills` ≠ 你安装的 Skills

`~/.codex/memories/skills/` 是记忆系统自己沉淀的流程包；  
`~/.codex/skills/` 是你（或插件）安装的正式 Skill。

**两套完全独立**，不会自动互相升级。记忆里的 skill 会随合并 Agent **增量更新**，但不是一次写完就冻结。

### 4. `thread/search` 是另一回事

IDE / 客户端可以用 `thread/search` 在**全部历史会话原文**里做关键词搜索（底层是 ripgrep）。

这和 Agent 用的**记忆手册**是两条线：

| | 记忆检索 | thread/search |
|--|---------|---------------|
| 搜什么 | 整理好的手册 | 原始会话 jsonl |
| 谁用 | 运行时 Agent | 客户端功能 |
| 向量 | 无 | 无 |

Agent 默认**没有**「搜全部历史会话」这个工具；它主要靠记忆手册 + 必要时沿手册里的路径去读原始会话。

### 5. 不是全文记住，是会话摘要 + 手册

超长会话进 Phase 1 时会**截断中间、保留头尾**，再让模型压缩成高信号笔记——不是把整段对话原样入库。

```
                          ↓

│  summary 注入 → 搜 MEMORY.md → 按需开摘要/skill/jsonl    │

```

## 一句话带走

Codex 记忆 = **SQLite 存笔记 → 文件夹是精选可读面 → Agent 先看 summary 再搜 MEMORY**；**Citation 的 rollout_ids** 把「哪场旧会话有用」写回数据库、影响下次 Top-256 排位；全程**文本搜索、无向量**。
